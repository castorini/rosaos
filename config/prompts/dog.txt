PuppyPi is a HiWonder quadruped robot dog exposed through ROS MCP.

This PuppyPi image is running the ROS 2 PuppyPi control stack through
rosbridge. The live robot is normally reachable from the laptop as
`raspberrypi.local:9090`.

Before moving, prefer read-only discovery tools such as `get_services`,
`get_topics`, `get_service_details`, and `get_topic_details` if you need to
confirm the currently available control surface.

Do not call `connect_to_robot`. The laptop-side dog MCP server is already
started by `scripts/start_dog_mcp.sh`, which connects it to PuppyPi rosbridge.
Calling `connect_to_robot` from an agent can reset the MCP server's rosbridge
connection and has caused worker TaskGroup/cancel-scope failures in rosaOS.

Safe setup / posture services:
```json
{
  "service_name": "/puppy_control/go_home",
  "service_type": "std_srvs/srv/Empty",
  "request": {},
  "timeout": 5
}
```

Enable or disable puppy control:
```json
{
  "service_name": "/puppy_control/set_running",
  "service_type": "std_srvs/srv/SetBool",
  "request": {
    "data": true
  },
  "timeout": 5
}
```

Run a predefined PuppyPi action group:
```json
{
  "service_name": "/puppy_control/runActionGroup",
  "service_type": "puppy_control_msgs/srv/SetRunActionName",
  "request": {
    "name": "<action-name>",
    "wait": true
  },
  "timeout": 15
}
```

Verified movement control uses `/puppy_control/velocity` with message type
`puppy_control_msgs/msg/Velocity`. For user requests such as "walk forward",
"move forward", "back up", "turn left", "turn right", or "rotate", use this topic with one bounded
`publish_for_durations` call, then send an all-zero stop message.

Verified forward movement request:
```json
{
  "topic": "/puppy_control/velocity",
  "msg_type": "puppy_control_msgs/msg/Velocity",
  "messages": [
    {
      "x": 5.0,
      "y": 0.0,
      "yaw_rate": 0.0
    },
    {
      "x": 0.0,
      "y": 0.0,
      "yaw_rate": 0.0
    }
  ],
  "durations": [
    0.8,
    0.5
  ]
}
```

Hiwonder's ROS 2 docs demonstrate forward walking with
`/puppy_control/velocity` using `x: 5.0`, not small fractional values. A live
test on June 1, 2026 confirmed that `x: 5.0` for 0.8 seconds moved the dog
forward. Earlier `x: 0.15` commands made it trot in place.

Rotation uses the same velocity topic. Do not use the
`/puppy/rotate_angle` action unless the user explicitly asks to test that
unverified action. For a turn, publish `x: 0.0`, `y: 0.0`, and a bounded
`yaw_rate`, then publish an all-zero stop message. Rotation works through
`yaw_rate`.

Raw velocity rotation tradeoff:

- `yaw_rate` around `0.2` gives a cleaner, more in-place turn, but it is not a
  small angle: `yaw_rate: 0.2` for 0.03 seconds produced roughly 135 degrees.
- Lower yaw rates produce smaller turns but arc more.
- Do not rely on sub-30ms durations for smaller angles; `yaw_rate: 0.2` for
  0.01 seconds still produced roughly 135 degrees, likely due to a
  controller/gait timing floor.
- Avoid `yaw_rate: 0.5` for small turns; even 0.01 seconds produced nearly a
  full rotation.

Calibrated cleaner rotation request:
```json
{
  "topic": "/puppy_control/velocity",
  "msg_type": "puppy_control_msgs/msg/Velocity",
  "messages": [
    {
      "x": 0.0,
      "y": 0.0,
      "yaw_rate": 0.2
    },
    {
      "x": 0.0,
      "y": 0.0,
      "yaw_rate": 0.0
    }
  ],
  "durations": [
    0.03,
    0.5
  ]
}
```

Use positive `yaw_rate` for a left / counterclockwise turn and negative
`yaw_rate` for a right / clockwise turn, unless live testing shows this
particular PuppyPi image has the signs reversed. Do not claim a specific angle
turned unless the user confirms it, a calibration table exists, or odometry is
available.

ROS 2 actions are present, but were not verified for movement through ROS MCP:

- `/puppy/move_distance`, likely type `puppy_control_msgs/action/MoveDistance`.
  Positive `distance` moves forward and negative `distance` moves backward.
- `/puppy/rotate_angle`, likely type `puppy_control_msgs/action/RotateAngle`.
- `/puppy/move_to_pose`, likely type `puppy_control_msgs/action/MoveToPose`.

The following action goal was accepted by rosbridge but did not physically move
the dog and timed out through MCP, so do not prefer it:
```json
{
  "action_name": "/puppy/move_distance",
  "action_type": "puppy_control_msgs/action/MoveDistance",
  "goal": {
    "distance": 10,
    "speed": 0.5
  },
  "timeout": 15
}
```

The connected PuppyPi's rosapi can list action names but may not expose enough
action metadata for automatic schema discovery. If action result reporting times
out, do not immediately retry movement. Confirm with the user whether the dog
moved.

The live ROS 2 graph also exposes `/puppy_control/velocity_move` with
`puppy_control_msgs/msg/Velocity`, plus `/cmd_vel` with
`geometry_msgs/msg/Twist`. Prefer `/puppy_control/velocity` because it was
physically verified.
```json
{
  "topic": "/puppy_control/velocity",
  "message_type": "puppy_control_msgs/msg/Velocity",
  "message": {
    "x": 5.0,
    "y": 0.0,
    "yaw_rate": 0.0
  }
}
```

- The camera topic is `/image_raw`; compressed variants are also available.
- Use ROS 2 message and service type names with `/msg/` and `/srv/`.
- For movement requests, prefer one bounded `publish_for_durations` call on
  `/puppy_control/velocity`, then stop. For a forward test, use the documented
  PuppyPi scale: `x: 5.0`, `y: 0.0`, `yaw_rate: 0.0` for no more than 1
  second, followed immediately by all-zero velocity for at least 0.5 seconds.
- For rotation requests, use `/puppy_control/velocity` with `x: 0.0`,
  `y: 0.0`, and a short bounded `yaw_rate`, followed immediately by all-zero
  velocity for at least 0.5 seconds. Follow the rotation tradeoff guidance
  above: `yaw_rate` around `0.2` is cleaner but large; lower yaw rates make
  smaller turns but arc more.
- Always stop by publishing zero velocity after any movement attempt.
- Do not claim a distance traveled unless you have real odometry or user
  confirmation. PuppyPi may step/trot in place if the gait or velocity scale is
  wrong.
- Do not assume the older `/puppy/move_distance_trigger` or
  `/puppy/rotate_angle_trigger` services exist; they were not present on this
  connected PuppyPi. The ROS 2 action names were present instead.
